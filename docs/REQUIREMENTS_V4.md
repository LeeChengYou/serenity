# Serenity Signal V4 — API Key 動態調度與繁中翻譯需求規格

> 版本：v1.0 | 建立：2026-07-05 | 整理者：Fable（監督者）
> 背景：使用者有 4 組 Gemini API key（各自獨立配額：RPM 每分鐘上限
> 與 RPD/每週上限）；曾發生單 key 429 導致批次中斷（A-4 首輪 22/30）。

---

## R4-1 API Key 池與任務親和調度

### Key 角色分配（任務親和 affinity）

| Key | 角色 | 負責任務 | 模型 | 特性 |
|-----|------|---------|------|------|
| KEY_1 | 互動主力 | AI 對話、dossier 經理人分析 | gemini-2.5-flash | 低延遲、使用者面前，量小 |
| KEY_2 | 批次生成 | 評分卡批次、未來批次任務 | gemini-2.5-flash | 爆量型，最常 429 |
| KEY_3 | 翻譯專用 | 新聞/貼文繁中翻譯 | gemini-2.5-flash-lite | 高量低價值，用最省的模型 |
| KEY_4 | 機動備援 | 任何任務 429 時的溢流 | 隨任務 | 平時閒置，保持配額滿 |

### 調度規則（server 內建 KeyManager）

1. **親和優先**：每類任務先用自己的主力 key。
2. **429 溢流**：主力 key 429 → 標記冷卻 → 換備援 KEY_4 →
   還是 429 → 換其他非冷卻 key → 全部冷卻則誠實回錯（zh-TW 訊息）。
3. **冷卻策略**：同 key 首次 429 冷卻 60 秒（RPM 型）；同 key
   10 分鐘內第 3 次 429 視為每日/每週配額耗盡，冷卻至下一個
   太平洋時間午夜。冷卻狀態存記憶體即可（重啟歸零，無害）。
4. **設定**：`.env` 讀 `GEMINI_API_KEY`（=KEY_1，向後相容）與
   `GEMINI_API_KEY_2/3/4`。缺任何一個都能跑（池自動縮小），
   **key 值由使用者自行填入 .env，程式碼與文件不得出現真實 key**。
5. **可觀測**：`GET /api/keypool` 回報各 key 目前狀態
   （遮罩顯示末 4 碼、冷卻中/可用、今日呼叫數、429 數）。

### 配額節約鐵律（所有任務通用）

- **批次合併**：翻譯一次送最多 20 條文本（單一 JSON 請求），
  不逐條呼叫。
- **快取優先**：翻譯結果、dossier、評分卡都先查 DB 快取，
  同內容絕不重複計費。
- **Live API 不用於此案**：Live/串流配額留給未來語音功能，
  文字翻譯一律用 flash-lite 文字模型。

## R4-2 繁中翻譯功能（保留英文原文）

### 後端

- 新表：`translations(src_hash text pk, src_text, translated_text,
  model, created_at)`——src_hash = sha256(原文)，同文永不重譯。
- `POST /api/translate`：
  ```
  請求:  {"texts": ["...", "..."]}        # 最多 20 條/次
  回應:  {"translations": ["...", "..."], "cached": [true,false,...],
          "error": null | "zh-TW 錯誤訊息"}
  ```
  - 先查快取；未命中的合併成單一 Gemini 呼叫（flash-lite，
    KEY_3 親和），提示詞要求：翻成台灣繁體中文、保留術語/
    股票代號/數字原樣、只回 JSON 陣列。
  - 部分失敗時：已快取的照常回，失敗的回 null，前端顯示原文。
- 翻譯**不自動預跑**（3,222 篇新聞全翻會燒光配額）——
  只翻使用者按按鈕時畫面上的內容。

### 前端

- 新聞面板與 X 貼文摺疊區各加一顆「譯 中/EN」切換按鈕：
  - 第一次按：收集當前可見條目的標題+摘要 → 一次 POST /api/translate
    → 顯示繁中，英文原文以較淡字體保留在下方（不是取代）。
  - 再按：切回純英文顯示。翻譯結果留在記憶體，切換不重打 API。
- 載入中顯示「翻譯中…」；API 失敗顯示「翻譯暫時不可用」+保留原文。

## 驗收原則

1. 任一 key 缺失/失效不擋任何核心功能（純數據照常）
2. 翻譯絕不覆蓋原文；快取命中不計費
3. 429 換 key 過程對使用者透明，全池耗盡時誠實提示
4. 文件與程式碼零真實 key；.env 由使用者自行維護
