# 判斷力手冊（JUDGMENT）

> 目的：把「需要品味的判斷」壓成可執行的判準。每條附一個正例、一個反例，
> 例子全部來自本 repo 真實事件。拿不準時：先對照本檔，仍拿不準 → 按第 3 節問使用者。

---

## 1. 何時該升級模型

**判準**：滿足任一 → 升級（路徑見 DISPATCH.md 第 4 節）：
- 同一子任務兩輪修復後測試仍紅。
- 任務需要「取捨」而非「執行」（兩個都對的方案選一個、規格有歧義要裁決）。
- 錯誤訊息看不懂且搜尋 repo/網路後仍無假設可測。

**正例**：sonnet 兩次修 arena 撮合邏輯都讓別的測試轉紅 → 帶兩次 diff + 測試輸出升 opus。
opus 發現根因是 fill 缺 `decided_date < as_of` 時間紀律防護——這是概念缺口，不是手滑。

**反例**：haiku 列檔案清單漏了一個目錄，於是「再給 haiku 一次機會」重跑同樣的 prompt。
錯。重試同 prompt 期待不同結果 = 燒 token；一次就升 sonnet 或自己用 Glob 查。

---

## 2. 何時算真的完成

**判準**：以下全部成立才能對使用者說「完成」：
- 驗收測試實跑，輸出數字貼在回報裡（arena = 70/70 + exit 0）。
- 改 server.py → server 重啟過且 curl 過受影響端點看到正確 payload。
- 改 dashboard/ → 明確告知使用者 Ctrl+F5（快取曾造成「功能消失」的誤報，ROADMAP K-2）。
- 涉及解析外部輸入（LLM 回應、網路 API）→ 至少一次真實冒煙，不能只靠 mock。
- 文件改動已本地 commit。

**正例**：V6 驗收 70/70 後，監督者仍跑了一次 live daily——抓到 Gemini 自創欄位名
導致 9 筆買單靜默消失。若當時以「測試全過」收工，這個 bug 會在無人看的排程裡吞單一個月。

**反例**：「我已修好翻譯按鈕，前端程式碼正確。」但沒驗證瀏覽器實際顯示、沒提快取。
使用者看到的仍是舊頁面 → 回報「沒修好」→ 來回浪費三輪。程式碼正確 ≠ 使用者看得到。

## 3. 何時該停下來問使用者

**判準**：滿足任一 → 停下來問，其餘情況自主前進不要問：
- 動作不可逆且非本次任務明示範圍（刪資料表、覆蓋 `data/serenity.sqlite`、force push）。
- 檔案內容與聊天決策矛盾（可能 OneDrive 回退）→ 查 git 分辨後仍矛盾就呈報，不擅自二選一。
- 要花使用者的錢或額度（批次打 Gemini、註冊排程改機器狀態）。
- 兩個合理方案對產品方向有長期影響（如「月結清倉 vs 持倉延續」層級的取捨）——
  給出你的建議+理由再問，不要開放式丟問題。

**正例**：REQUIREMENTS_V6.md 的 R6-1 被外力改回舊版兩領域文字。正確處置：不擅自改回，
回報使用者「檔案與定案矛盾，疑 OneDrive 回退，是否照三領域修正？」

**反例**：「要不要我現在跑測試？」「需要我繼續嗎？」——這種問題永遠不問。
可逆、在任務範圍內 → 直接做完再回報。問了只是把工作變成使用者的負擔。

## 4. 什麼訊號代表方向錯了（該換路，不是再重試）

**判準**：出現任一 → 停止當前路線，寫下目前假設，換方法或升級：
- 兩輪不同的修法後同一測試仍紅。
- 修法開始「關掉驗證/放寬斷言/加特例」而不是解釋根因。
- 每修一處就冒出新的紅（打地鼠）——通常是抽象層選錯了。
- 只有 mock 過、真實輸入永遠過不了。

**正例**：J-10 排程下 pending 單永不成交。錯的路：調撮合重試邏輯。對的路：質疑輸入——
`as_of` 預設用了日曆日（2026-07-06）但價格表只到 2026-07-02，台北早上跑排程時
「今天」永遠沒有價格。根因在預設值，不在撮合。

**反例**：Gemini 回應解析失敗，於是把驗證改寬鬆「能塞就塞」。錯。正解是釘死 schema
＋畸形單顯式記拒單。放寬驗證 = 把 bug 從看得見改成看不見。

## 5. 品質底線怎麼驗（本 repo 特化）

**判準**（出貨前逐條核）：
- 每個對外數字可追溯 SQLite 真實列（抽 2–3 個數字手動 SELECT 對）。
- 時間紀律：任何用到日期的查詢確認 `<= as_of`；撮合類確認 `decided_date < as_of`。
- 冪等：同指令跑兩次，第二次不產生重複列、不報錯。
- 解析 LLM/外部輸出的每條路徑：畸形輸入 → 顯式記錄（拒單/告警），絕不靜默丟棄。
- 使用者可見文字是 zh-TW；金融內容帶「模擬資金・非投資建議」性質聲明。
- 樣本不足（n<30）的結論標 insufficient，不下定論。

**正例**：拉回策略 n=7 時標「insufficient，8 月中樣本夠再下結論」——這是本產品的信譽核心。

**反例**：為了讓評分卡好看，缺基本面欄位時填產業平均值。這違反鐵律第 1 條，
等同任務失敗，即使 UI 變漂亮了。

## 6. Harness 的極限（誠實條款）

拆解、驗證、fresh-context 審查、多答案評審能補「執行品質」，補不了以下兩類：
- **模糊題**（規格本身沒定義對錯，如「這個 UI 風格好不好」）：不要假裝有客觀答案。
  做法：給 2–3 個具體選項＋各自 trade-off，讓使用者選；或 AskUserQuestion 附 preview。
- **品味判斷**（策略取捨、文案語感）：sonnet 等級的第二意見只能防明顯錯誤，
  防不了平庸。做法：明說「這是品味判斷，我的建議是 X，理由是 Y，你可以否決」。
  重大者建議使用者用更強的模型或人工複核。查不到的事實就標「未查證」，不要編。
