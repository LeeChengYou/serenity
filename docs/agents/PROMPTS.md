# 派工 Prompt 模板（PROMPTS）

> 用法：主對話派 Agent 時，複製對應模板，把 `{…}` 填滿。**每個空格都要填**——
> 空著的驗收條件 = 沒有驗收。model 選擇見 DISPATCH.md 第 3 節。
> 每個模板已內建本 repo 鐵律與環境陷阱，不要刪這些段落。

## 共用尾段（每個模板末尾都要附上）

```
【鐵律】零捏造數據（缺=NULL/insufficient）；零 look-ahead（資料 <= as_of）；
機密不落輸出（.env / x_curl/ / data/browser_profile/）；測試用 StubBackend，不打真 Gemini。
【環境】DB 是 data/serenity.sqlite（不是 app.db）；跑 python 前設 PYTHONIOENCODING=utf-8；
PowerShell 5.1 沒有 &&，含引號的 one-liner 用 Bash 工具；搜尋排除 .claude/worktrees/。
【回報合約】只回：結論（≤10 行）+ 改動/發現的檔案清單（檔案:行號）+ 驗收條件逐條的
實際結果（貼指令輸出原文）。長產物寫進檔案回報路徑。做不到的項目明說做不到＋原因，
不要含糊帶過。
```

---

## 1. 搜尋（subagent_type: Explore，model: haiku 簡單/sonnet 一般）

```
【目標】找出 {什麼東西：定義/呼叫點/資料流}，因為主對話接下來要 {動機：改 X/評估 Y}。
【範圍】從 {起點目錄或檔案，如 scripts/} 開始；排除 .claude/worktrees/。
搜尋廣度：{medium / very thorough}。
【只讀不改】你是唯讀偵察，不 Edit 任何檔案。
【驗收條件】
- 每個發現附 檔案:行號 與一句話說明其角色。
- 明確回答：{要回答的具體問題，如「fill 邏輯在哪個函式、由誰呼叫、日期條件是什麼」}。
- 沒找到就回報「未找到」＋已排除的地點清單，不要猜。
```

## 2. 實作（subagent_type: general-purpose，model: sonnet，isolation: "worktree"）

```
【目標】實作 {功能}，動機：{為什麼要}。規格見 {docs/xxx.md 或內嵌規格}。
【介面契約】{函式簽名/API payload/資料表 schema——先由主對話定死，不留給 agent 發明}
【禁區】不改 {不該碰的檔案}；不新增外部依賴除非規格明列。
【驗收條件】（全部通過才算完成）
- [ ] {測試指令} 輸出 {期望結果，如 70/70、exit 0}
- [ ] python -m py_compile {改動的 .py 檔} 通過
- [ ] {功能級驗證，如：curl http://127.0.0.1:8787/api/xxx 回傳含欄位 a,b,c}
- [ ] 同指令重跑第二次冪等（無重複列、無報錯）
【自驗要求】每個驗收條件貼實際輸出原文；沒跑就標「未跑」＋原因。
```

## 3. 重構（subagent_type: general-purpose，model: sonnet，isolation: "worktree"）

```
【目標】重構 {對象}，動機：{可讀性/去重/為 X 鋪路}。**行為不得改變。**
【基準】重構前先跑 {測試指令} 並記錄輸出作為基準；重構後輸出必須完全一致。
【範圍】只動 {檔案清單}；公開介面（函式簽名/API shape/DB schema）不得變動，
若發現不變動做不到 → 停下回報，不要擅自改介面。
【驗收條件】
- [ ] 重構前後 {測試指令} 輸出一致（兩份都貼）
- [ ] git diff --stat 貼出（確認沒動到範圍外的檔）
- [ ] 淨行數變化與動機一致（去重卻變長 → 說明原因）
```

## 4. 研究（subagent_type: general-purpose，model: sonnet；判斷型結論用 opus）

```
【目標】回答 {具體問題}，用於 {決策：選方案/評估風險}。
【來源優先序】官方文件 > 原始碼 > 一手數據 > 部落格。每個結論附來源（URL 或 檔案:行號）。
【誠實要求】查不到就寫「未查到」；來源互相矛盾就並列矛盾點；
不確定度標示：確定 / 很可能 / 猜測。禁止填補式編造。
【驗收條件】
- 回答 {子問題 1}、{子問題 2}…每題有來源。
- 結尾給一段「如果只能記住一件事」的摘要（≤3 句）。
- 產出寫入 {scratch/research_xxx.md 或指定路徑}，回報路徑。
```

## 5. 審查（subagent_type: general-purpose，model: opus；必須 fresh context——不能由實作者自審）

```
【目標】審查 {分支/diff/檔案}，這是 {誰} 做的 {什麼}，驗收清單如下。
你沒有參與實作，請以懷疑視角逐條核驗，不要信任實作者的自述。
【核驗方法】
- 檔案類：實際 Read 檔案確認存在、完整、內容與宣稱一致。
- 程式碼類：實跑 {測試指令}，貼輸出；抽查 2–3 個關鍵路徑的邊界（空輸入/畸形輸入/重跑冪等）。
- 數據類：抽 2–3 個數字用 sqlite3 SELECT 對回 data/serenity.sqlite。
【特別檢查】（本 repo 慣性弱點）
- 解析 LLM/外部輸出處：畸形輸入是否顯式記錄而非靜默丟棄？
- 日期邏輯：是否全部 <= as_of？撮合類是否 decided_date < as_of？
- zh-TW 使用者可見文字；免責聲明未被刪。
【回報】三段式：❌ 必修（違反鐵律/會壞）→ ⚠️ 建議（風險/債）→ ✅ 已核驗通過的項目。
每條附 檔案:行號 與證據。整體結論：通過 / 退回（附退回原因排序）。
```
